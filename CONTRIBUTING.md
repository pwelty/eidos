# Contributing to Eidos

Thank you for your interest in contributing to the Eidos specification! This document provides guidelines for contributing to this open-source project.

## How to Contribute

### Reporting Issues

If you find a problem with the specification or have suggestions for improvement:

1. **Check existing issues** first to avoid duplicates
2. **Create a detailed issue** with:
   - Clear title and description
   - Steps to reproduce (if applicable)
   - Expected vs actual behavior
   - Relevant specification sections
   - Proposed solutions (if any)

### Suggesting Enhancements

We welcome suggestions for improving the Eidos specification:

- **New patterns** or best practices
- **Framework implementations** for FRAMEWORKS.md
- **Clarifications** to existing documentation
- **Examples** and use cases

### Making Changes

1. **Fork the repository** and create a feature branch
2. **Make your changes** following our style guide
3. **Test your changes** (ensure all links work, examples are valid)
4. **Submit a pull request** with a clear description

## Style Guide

### Documentation Standards

- **Clear and concise** writing
- **Consistent terminology** throughout
- **Framework-agnostic** language in core specs
- **Specific implementations** belong in FRAMEWORKS.md
- **Examples** should be practical and realistic

### File Organization

- **Core specifications**: database.md, ui.md, engine.md, api.md, foundations.md
- **Implementation details**: FRAMEWORKS.md
- **Project meta**: README.md, CONTRIBUTING.md, LICENSE

### Writing Style

- Use **active voice** when possible
- Write in **present tense**
- Use **bullet points** for lists and requirements
- Include **code examples** for clarity
- Add **diagrams** for complex concepts

## Types of Contributions

### 🐛 Bug Fixes
- Typos and grammatical errors
- Broken links or references
- Inconsistencies in the specification

### 📝 Documentation Improvements
- Clarifying existing content
- Adding missing information
- Improving examples and code snippets

### 🚀 New Features
- New architectural patterns
- Additional framework implementations
- Extended examples and use cases

### 🔧 Framework Implementations
- New technology stacks in FRAMEWORKS.md
- Updated framework examples
- Best practices for specific tools

## Pull Request Process

1. **Create a descriptive title** for your PR
2. **Reference any related issues** in the description
3. **Explain your changes** and why they're needed
4. **Update documentation** if you're changing behavior
5. **Ensure all links work** and examples are valid

### PR Template

```markdown
## Description
Brief description of what this PR does.

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Framework implementation

## Testing
- [ ] All links verified
- [ ] Examples tested
- [ ] Documentation reviewed

## Related Issues
Fixes #(issue number)
```

## Framework Implementation Guidelines

When adding framework-specific implementations to FRAMEWORKS.md:

### Requirements
- **Working examples** with proper syntax
- **Consistent patterns** across similar frameworks
- **Complete implementations** (not partial snippets)
- **Comments** explaining framework-specific choices

### Structure
- Follow existing section organization
- Use appropriate code blocks with language highlighting
- Include setup/configuration examples
- Provide links to official documentation

### Technologies We're Interested In
- **Frontend**: React, Vue, Angular, Svelte, Next.js, Nuxt, etc.
- **Backend**: FastAPI, Django, Rails, ASP.NET Core, Spring Boot, etc.
- **Databases**: PostgreSQL, MySQL, MongoDB, Supabase, etc.
- **Languages**: TypeScript, Python, C#, Ruby, Java, Go, Rust, etc.

## Community Guidelines

### Be Respectful
- Use inclusive language
- Be constructive in feedback
- Respect different approaches and opinions
- Help create a welcoming environment

### Be Collaborative
- Share knowledge and experience
- Ask questions when unclear
- Offer help to other contributors
- Participate in discussions constructively

### Be Professional
- Keep discussions focused on the project
- Use appropriate language and tone
- Provide evidence for claims and suggestions
- Accept feedback gracefully

## Getting Help

If you need help contributing:

- **Create an issue** with the "question" label
- **Start a discussion** in the repository
- **Review existing issues** and discussions
- **Check the specification** for context and examples

## Recognition

All contributors are recognized in the project. We maintain a list of contributors and their contributions.

### Types of Recognition
- **GitHub contributors page** for code contributions
- **Acknowledgments** in documentation
- **Community highlights** for significant contributions

## Development Environment

No special development environment is required for documentation contributions:

- **Text editor** of your choice
- **Markdown knowledge** (helpful but not required)
- **Git and GitHub** familiarity
- **Basic command line** usage

## Questions?

If you have questions about contributing, please:

1. Check existing issues and discussions
2. Create a new issue with the "question" label
3. Be as specific as possible about what you need help with

Thank you for helping improve the Eidos specification! 🚀
